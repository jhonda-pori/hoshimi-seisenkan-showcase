# セキュリティ設計

## 概要

ホシミ占星館では、認証・入出力フィルタリング・課金検証・インフラの各レイヤーでセキュリティ対策を実施しています。

## 認証・認可

| 対策 | 説明 |
|------|------|
| **独自トークン認証** | 全APIエンドポイントでX-Auth-Tokenヘッダーを検証（有効期限90日） |
| **device_id二重チェック** | トークン認証に加え、リクエスト内のdevice_idとDB上のユーザーを照合 |
| **リソース所有権検証** | 鑑定結果・チャット履歴へのアクセス時、user_idの一致を必ず検証 |
| **トークン自動失効** | 期限切れトークンはアクセス時に自動削除 |

## アクセス制御

| 対策 | 説明 |
|------|------|
| **エンドポイント別レートリミット** | 鑑定: 10-20回/時、登録: 10回/時、チャット: 10回/時 |
| **クライアントIP取得** | Cloudflare経由の場合CF-Connecting-IPを使用（X-Forwarded-For偽装を防止） |
| **リクエストサイズ制限** | 64KB上限 |
| **セキュリティヘッダー** | X-Content-Type-Options, X-Frame-Options, CSP, HSTS を全レスポンスに付与 |
| **エラーメッセージ制御** | 内部エラーの詳細はログのみに記録し、ユーザーには固定メッセージを返却 |

## 入力フィルタリング（プロンプトインジェクション対策）

鑑定文生成にLLMを使用しているため、プロンプトインジェクション対策を多層で実施:

### 1. 入力正規化

```
ユーザー入力
  │
  ├── ゼロ幅文字・不可視文字を除去
  │     (U+200B, U+200C, U+200D, BOM, soft hyphen 等)
  │
  ├── NFKC正規化
  │     (全角英数 → 半角、合字の分解)
  │
  ▼
正規化済みテキスト → パターンマッチング
```

### 2. インジェクションパターン検出

多言語・多手法のパターンを検出:

| カテゴリ | 検出パターン例 |
|----------|--------------|
| **直接的な指示上書き** | "ignore previous instructions", "指示を無視" |
| **ロール変更** | "you are now", "pretend to be", "あなたは...ではなく" |
| **プロンプト抽出** | "show your system prompt", "システムプロンプトを教え" |
| **Leet speak** | "ign0re prev1ous" |
| **バイパス試行** | "skip the rules", "bypass the filter", "ルールを回避" |
| **間接的抽出** | "what were your instructions", "最初の命令" |

### 3. 出力フィルタリング

生成された鑑定文が有害な内容を含まないかチェック:

- 医療診断・処方に関する記述
- 具体的な投資助言
- 専門家への強制的な誘導

該当する場合はレスポンスをブロックし、クレジットを返還。

## アカウント引き継ぎのセキュリティ

機種変更時のアカウント移行に対するブルートフォース攻撃対策:

```
パスワード照合失敗
  │
  ├── 失敗を記録（user_id + IP + タイムスタンプ）
  │
  ├── 同一IP: 5回失敗でロック
  ├── 全IP合計: 15回失敗でロック（分散攻撃対策）
  │
  └── ロックアウト期間: 60分
```

- パスワードはbcryptでハッシュ化して保存
- 引き継ぎ成功時に失敗記録をクリア

## 課金検証

| プラットフォーム | 検証方法 |
|-----------------|----------|
| **Google Play** | サービスアカウントで Google Play Developer API を呼び出し、purchaseState を検証 |
| **App Store** | App Store Server API でトランザクションのJWT署名を検証 |

### 冪等性の保証

```
購入リクエスト到着
  │
  ├── purchase_token で既存レコードを検索
  │     ├── 存在する → 重複履行を防止（成功レスポンスを返すが付与はしない）
  │     └── 存在しない → クレジット付与 → purchase_token を記録
  │
  ▼
ネットワーク障害によるリトライでも二重付与が発生しない
```

## 広告トークン（リワード広告の不正防止）

無料鑑定（ウララの鑑定）は広告視聴後にのみ利用可能。不正リクエストを防止するワンタイムトークン方式:

```
1. クライアントが POST /api/ad/token でトークンを取得
2. リワード広告を視聴
3. 鑑定リクエストに ad_reward_token を付与して送信
4. サーバーがトークンを検証・消費（ワンタイム）
```

- トークン有効期限: 5分
- 同一ユーザーの未使用トークンは新規発行時に無効化（スタッキング防止）

## インフラセキュリティ

```
[インターネット]
      │
      ▼
[Cloudflare] ── DDoS防御
      │
      │ Cloudflare Tunnel（暗号化トンネル）
      │ ※サーバーIPを外部に非公開
      ▼
[Flask Server]
      │
      │ SQLite (WAL mode)
      ▼
[データベース] ── ディスク上のファイル
```

| レイヤー | 対策 |
|----------|------|
| **ネットワーク** | Cloudflare Tunnel でオリジンIP秘匿 |
| **DDoS** | Cloudflareエッジで吸収 |
| **アプリケーション** | Flask-Limiter によるエンドポイント別レートリミット |
| **データベース** | SQLite WALモードで読み書きの並行性確保 |
| **ログ** | リクエストIP・メソッド・パス・ステータスコードを記録 |
